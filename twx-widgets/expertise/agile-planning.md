## Skill reference

The implementation skill is `twx-widgets` — it handles all widget coding, debugging, and packaging.
When writing US-PLAN.md tasks, the executor will have access to this skill's workflows, references, and templates.

## Scope estimation adjustments

Widget development has specific complexity factors that affect story point estimation:

| Factor | Impact on estimation |
|--------|---------------------|
| New project scaffold (webpack, tsconfig, package.json) | +2 points — many files, easy to get wrong |
| CombinedExtensions.js constraint (lazy loading) | +1 point if external libs involved |
| Property binding (static + bound + order-independent) | +1 point per complex property interaction |
| Dual lifecycle (IDE Composer + Runtime) | +1 point — two separate code paths to implement |
| Vendor splitting (React/MUI shared across widgets) | +2 points — webpack config complexity |
| Multi-widget project (adding to existing) | +1 point — must not break existing widgets |
| Chrome DevTools debugging | 0 — handled by MCP, no extra effort |

**Rule of thumb:** A simple widget modification = 1-2 points. A new widget from scratch = 3-5 points. A new project with vendor splitting = 5-8 points (consider splitting into multiple stories).

## Key acceptance criteria patterns

When writing Given/When/Then for widget stories, always include these domain-specific criteria where applicable:

### CombinedExtensions.js size
```
Given the widget extension is built
When CombinedExtensions.js is generated by ThingWorx
Then the runtime bundle contributes < 50KB to CombinedExtensions.js
And external libraries are loaded via dynamic import() as separate chunks
```

### Property binding resilience
```
Given the widget has bound properties
When afterRender() fires before property setters (or vice versa)
Then the widget renders correctly regardless of call order
And the widget handles partial state (missing data) gracefully
```

### IDE/Runtime parity
```
Given the widget is added to a mashup in Composer
When the user configures properties in the IDE
Then the IDE preview reflects the configuration accurately
And the Runtime renders the same result with live data
```

### Build integrity
```
Given the widget source code is modified
When npm run build is executed
Then the build completes without errors
And the .zip extension package is generated in target/
```

## Story splitting guidance

### Split by lifecycle when complex
- **Story A:** IDE (Composer) — property panel, preview rendering, validation
- **Story B:** Runtime — data binding, event handling, live rendering

### Split by concern for new projects
- **Story 1:** Project scaffold (webpack, tsconfig, package.json, build pipeline)
- **Story 2:** Widget implementation (IDE + Runtime code)
- **Story 3:** External library integration (lazy loading, vendor splitting if needed)

### Do NOT split
- Simple property additions (1 property = 1 story including IDE + Runtime)
- Bug fixes (fix + verify in one story)
- Style-only changes (SCSS modifications)

## Architecture constraints for plans

When creating US-PLAN.md tasks, the executor must know:

1. **Read before write** — Every task that generates widget code MUST start by reading an existing example project in `~/Documents/01_Sources/01_Databox/01_Widgets/`. This is non-negotiable — the skill enforces it.

2. **Webpack-based, not Ant** — Standard ThingWorx packaging docs don't apply. The build uses a custom webpack pipeline with specific plugins (metadata generator, source URL updater, upload).

3. **TypeScript preferred** — New widgets should use TypeScript with `typescriptwebpacksupport` decorators unless there's a specific reason not to.

4. **publicPath matters** — Webpack output `publicPath` must be `/Thingworx/Common/extensions/{package}/ui/{widget}/` for chunk loading to work at runtime.

5. **No webpack chunks in CombinedExtensions** — Only the entry point goes into CombinedExtensions.js. Libraries must use dynamic `import()` to become separate chunks loaded on demand.

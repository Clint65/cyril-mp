# User Story: Documentation des pièges connus

## Story

**As a** skill NL→SQL
**I want** connaître les pièges spécifiques du schéma Databox avec des exemples concrets
**So that** je ne génère jamais de SQL silencieusement faux

## Acceptance Criteria

### AC1: Piège end_date documenté
- **Given** le document des pièges
- **When** le piège "oubli de end_date IS NULL" est consulté
- **Then** il montre la requête fausse (résultats multipliés), la requête correcte, et l'impact chiffré

### AC2: Piège aliases id_mapping documenté
- **Given** le document des pièges
- **When** le piège "alias manquant sur id_mapping" est consulté
- **Then** il montre l'erreur SQL produite et la correction avec aliases distincts

### AC3: Piège cast TEXT documenté
- **Given** le document des pièges
- **When** le piège "jointure article sans cast" est consulté
- **Then** il montre la jointure silencieusement vide et la correction avec ::text

### AC4: Piège invoices_sign documenté
- **Given** le document des pièges
- **When** le piège "avoirs comptés positivement" est consulté
- **Then** il montre le CA gonflé sans le signe et le CA correct avec

### AC5: Les 6 pièges critiques couverts
- **Given** les 6 règles critiques de la spec
- **When** le document est finalisé
- **Then** chaque règle a son piège documenté avec avant/après

## Tasks

1. [ ] Rédiger chaque piège avec requête fausse et requête correcte
2. [ ] Valider par exécution sur la base quand possible
3. [ ] Écrire references/exemples/pieges_connus.md

## Technical Notes

- Les 6 pièges critiques : end_date IS NULL, résolution idcor, cast TEXT, aliases id_mapping, invoices_sign, décodage value_lists
- Format : nom du piège → symptôme → requête fausse → requête correcte → explication

## Definition of Done

- [ ] All acceptance criteria met
- [ ] 6 pièges documentés avec avant/après
- [ ] Fichier pieges_connus.md créé
